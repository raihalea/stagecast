# ADR 0003: 障害時フェイルオーバーと配信途中のリソース再起動方針

- ステータス: Accepted
- 日付: 2026-06-14
- 関連: `DESIGN.md` 7 章・9.1、ADR 0001（D-6 ECS, D-7 Valkey）、ADR 0002（字幕バス）

## コンテキスト

`DESIGN.md` 9.1 は「障害時のフェイルオーバーと、配信途中のリソース再起動方針」を実装フェーズの
詳細設計事項として残している。本 ADR で、配信中にコンポーネントが落ちたときの復旧方針と、
配信を止めない（または最短で再開する）ための設計を確定する。

配信は **イベント単位の独立スタック**（SFU/Egress/字幕ワーカー + Valkey, ADR D-6/D-7）で構成され、
最大 3 イベントが並行する（N-5, 7.3）。常時稼働するのは制御層（Lambda/DynamoDB/Cognito/S3/CloudFront）のみ（N-1）。

### 目標（イベント単位）

| 指標               | 目標                                                                                      |
| ------------------ | ----------------------------------------------------------------------------------------- |
| 制御層（API/設定） | 実質無停止（Lambda 自動リトライ・DynamoDB 多AZ）                                          |
| SFU/Egress（映像） | 単一タスク障害から **60 秒以内**に再送出再開（YouTube は再接続を許容）                    |
| 字幕ワーカー       | **10 秒以内**に再起動し、Valkey Streams のバックログから追いつき（字幕欠落を最小化, N-2） |
| 確定字幕・録画     | **損失ゼロ**（確定字幕は S3 保存、録画は逐次アップロード）                                |

## 決定

### D-1. 制御層はステートレス・冪等で自己回復

- 制御 API は Lambda。失敗は API Gateway/クライアントのリトライで吸収。状態は DynamoDB（多AZ・
  オンデマンド）に永続化し、Lambda 自身は状態を持たない。
- イベントのライフサイクル状態（draft/scheduled/live/ended）は DynamoDB が真実源。プロセス
  再起動後もここから復元できる。

### D-2. オーケストレータの調整ループ（reconciliation）

- `media-orchestrator` は **望ましい状態（DynamoDB の live イベント集合）** と **実際の状態
  （CloudFormation スタック / ECS サービスの存在と健全性）** を定期照合する調整ループを持つ。
  - live なのにスタックが無い/失敗 → 再作成（`CloudFormationMediaStackProvisioner.provision`）。
  - ended なのにスタックが残存 → 破棄（`destroy`）。
  - スタックが `*_FAILED/ROLLBACK` → 一度破棄してから再作成（クリーン再構築）。
- これにより、オーケストレータ自身が再起動しても、ループが次回実行で齟齬を収束させる
  （自己修復）。`startEvent` は冪等（ADR フェーズ3 実装済み）なので二重起動は安全。

### D-3. メディア層（SFU/Egress）の再起動と RTMP 再接続

- ECS/Fargate サービスの **desiredCount と healthCheck** によりタスク障害は自動置換される
  （`EventMediaStack` は circuitBreaker と minHealthyPercent=0 で素早い置換を許容）。
- Egress（合成・RTMP 送出）が落ちた場合:
  1. タスクが再起動し、Valkey から **現在の発表状態（PresentationState）** を読んでレイアウトを
     復元する（発表状態は Valkey が保持。ADR D-7）。
  2. YouTube へは **同一ストリームキーで RTMP 再接続**する。YouTube Live は短時間の再接続を許容
     するため、視聴側は数秒の中断で復帰する。
  3. 録画は **S3 マルチパート/セグメント**で逐次アップロードし、再起動後は新セグメントを継続。
     既にアップロード済みの区間は失われない（N-4）。
- SFU(LiveKit) 障害時は参加者クライアントが自動再接続（LiveKit SDK の再接続）。発表者の
  publish 状態は再 publish で復帰する。

### D-4. 字幕ワーカーの再起動と Valkey Streams 追いつき（ADR 0002 と連動）

- 字幕ワーカーが落ちた場合、ECS が再起動する。字幕バスを **Valkey Streams**（ADR 0002）に
  した構成では、ワーカーは **最後に処理したストリーム ID** から `XREADGROUP` で再開でき、
  保持期間内のメッセージを取りこぼさない。
- 出力先（Sink）は **At-least-once** を前提に冪等であること。字幕は同一 (speakerId,startMs,language)
  で上書き表示できるため、重複配信は実害がない。確定字幕の S3 保存も同キー上書きで冪等。
- 独自字幕 API の視聴クライアントは再接続後に `subscribe` し、`CaptionConnectionHub` の
  **バックログ再送**で直近の確定字幕に追いつく（実装済み）。

### D-5. 配信途中のリソース再起動の原則

- **状態は外部化**（DynamoDB=設定/ライフサイクル、Valkey=ルーム/発表状態+字幕ストリーム、
  S3=録画/確定字幕）。各タスクはステートレスに保ち、いつ再起動しても外部状態から復元する。
- **冪等な起動/送出**：provision・字幕送出・録画アップロードはすべて冪等または再開可能。
- **部分障害の隔離**：イベントスタックは独立（N-5）。あるイベントの障害は他へ波及しない。
  字幕バスはイベント単位ストリームで隔離（ADR 0002）。

## 監視・検知

- ECS サービス/タスクの healthCheck、CloudWatch アラーム（タスク異常終了・RTMP 切断・字幕遅延）。
- オーケストレータ調整ループは EventBridge スケジュール（例: 30〜60 秒間隔）または
  CloudFormation/ECS イベント駆動で起動する。

## 結果（トレードオフ）

- **利点**: 外部化状態 + 冪等 + 調整ループにより、ほぼすべてのコンポーネントが「再起動で復旧」
  に統一され、運用が単純。イベント隔離で爆発半径を限定。
- **留意**: YouTube 再接続中の数秒の中断は許容範囲とする（完全無停止は二重 Egress 等が必要で
  コスト過大、N-1 と相反）。At-least-once に伴う字幕重複は Sink の冪等性で吸収する。

## 実装メモ（本 ADR に紐づくコード上の前提）

- 調整ループの骨格は `MediaOrchestrator`（冪等 start/stop）と
  `CloudFormationMediaStackProvisioner`（作成/完了待ち/失敗検知/削除）で用意済み。
  定期実行のスケジューラ（EventBridge → Lambda → reconcile）が今後の配線対象。
- 字幕の追いつきは `ValkeyStreamsCaptionBus`（ADR 0002）と `CaptionConnectionHub` のバックログで
  実装済み。ワーカーのコンシューマグループ ack/カーソル永続化は実結線時の対象。

## 未解決（将来）

- リージョン障害に対する DR（マルチリージョン）は本 ADR の射程外。
- 二重 Egress による完全無停止が必要な高 SLA 配信向けオプション。
