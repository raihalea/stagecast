# ADR 0002: 字幕バスの分散化（Valkey Streams 採用）

- ステータス: Accepted
- 日付: 2026-06-14
- 関連: `DESIGN.md` 6 章・9.1、ADR 0001（D-7 Valkey, D-8 字幕バス）

## コンテキスト

字幕バス（`DESIGN.md` 6 章）は、各エンジンが生成した字幕イベントを共通形式で受け取り、
各出力先（Sink）へ配る内部メッセージ経路である。現在は単一プロセス内の
`InProcessCaptionBus`（`CaptionBus` インターフェースのプロセス内実装）で動作している。

`DESIGN.md` 9.1 は「字幕バスの実装方式（メッセージング基盤）と暫定・確定字幕の同期方式」を
実装フェーズの詳細設計事項として残している。本 ADR でこれを確定する。

### 要件・制約

- **低遅延**: 字幕遅延は 3 秒以内が目標（N-2）。バス通過のオーバーヘッドは数十 ms に抑えたい。
- **イベント単位の隔離**: 最大 3 イベントが同時並行し、相互に干渉しない（N-5, 7.3）。
  バスもイベント単位で独立させる。
- **暫定/確定の両方を流す**: 独自字幕 API は暫定・確定の双方を配信する（6.3.2）。順序が重要。
- **複数 Sink へのファンアウト**: YouTube（確定・1言語）と独自 API（多言語）など、
  同一ストリームを複数コンシューマが読む。
- **再接続・追いつき**: 独自字幕 API の視聴クライアント再接続に備え、直近イベントの再生
  （バックログ）ができると望ましい（既に `CaptionConnectionHub` がアプリ層で実装）。
- **コスト**: メディア層・翻訳層は配信時のみ起動しゼロスケールする（N-1）。バス基盤も
  イベント時のみ課金が増える形が望ましい。常時稼働の固定費を増やさない。
- **既存資産の活用**: イベント単位メディアスタックは既に **ElastiCache for Valkey
  (Serverless)** を持つ（ADR D-7, `EventMediaStack`）。

## 選択肢

| 方式                                       | 遅延          | 隔離                    | コスト/運用                      | 評価                    |
| ------------------------------------------ | ------------- | ----------------------- | -------------------------------- | ----------------------- |
| **Valkey Streams**（既存 Valkey を再利用） | 低（〜数 ms） | キー/ストリーム名で容易 | 追加リソース不要・イベント時のみ | ◎                       |
| Amazon Kinesis Data Streams                | 中（200ms〜） | ストリーム/シャード単位 | 常時 or オンデマンド課金・運用増 | △（遅延/コスト）        |
| Amazon MSK (Kafka)                         | 低〜中        | トピック単位            | 常時クラスタ・運用重い           | ✕（N-1 に反する固定費） |
| Amazon SNS+SQS                             | 中            | トピック/キュー単位     | サーバレスだが順序/低遅延に弱い  | △（順序・遅延）         |
| EventBridge                                | 中〜高        | バス単位                | サーバレス                       | ✕（遅延・スループット） |

## 決定

**字幕バスの分散実装に Valkey Streams（Redis Streams 互換）を採用する。**

- `EventMediaStack` が既にイベント単位で持つ **ElastiCache for Valkey (Serverless)** を
  そのまま字幕バスにも用いる。追加の常時稼働リソースを作らない（N-1, 7.2）。
- ストリーム名は **イベント単位で名前空間化**する: `stagecast:{eventId}:captions`。
  これによりイベント間の干渉を防ぐ（N-5, 7.3）。共有状態のキー規約（ADR D-7）と一貫させる。
- 1 メッセージ = 1 字幕イベント（共通形式 6.1 を JSON 直列化）。`XADD` で追記し、
  各 Sink は **コンシューマグループ**（`XREADGROUP`）または独立カーソル（`XREAD`）で購読する。
  暫定/確定はメッセージの `status` フィールドで区別し、**ストリームの追記順序がそのまま
  配信順序**になるため暫定→確定の同期が自然に保たれる。
- **インターフェースは変更しない**。`packages/shared` の `CaptionBus`（`publish` / `subscribe`）
  をそのまま実装した `ValkeyStreamsCaptionBus` を追加し、`InProcessCaptionBus` と差し替え可能に
  する（ADR D-8 の方針どおり）。単一タスク内で完結する構成では引き続き `InProcessCaptionBus`
  を使ってよい（プロセス内なら最小オーバーヘッド）。
- **追いつき/再接続**: ストリームは一定期間（例: 直近 5 分 or `MAXLEN ~ N`）保持し、
  再接続したコンシューマは最後に処理した ID から再開できる。視聴クライアント向けの追いつきは
  アプリ層の `CaptionConnectionHub` バックログと併用する。
- **保持/上限**: `XADD ... MAXLEN ~ <N>` で近似トリミングし、メモリを抑える。確定字幕の
  永続化は引き続き S3（`CaptionStore`, 6.4）が担い、ストリームは短期バッファに徹する。

### いつ分散版を使うか

- **同一タスク内**（エンジンと Sink が同一プロセス）: `InProcessCaptionBus`（既定）。
- **クロスプロセス/クロスタスク**（例: 字幕ワーカーと独自字幕 API サーバを別タスクにする、
  Sink を水平スケールする）: `ValkeyStreamsCaptionBus`。

## 実装方針

- `services/caption-pipeline` に `ValkeyStreamsCaptionBus` を追加（本 ADR と同時に最小実装）。
- Valkey/Redis クライアントには直接依存せず、最小操作を `CaptionStreamClient`
  （`xadd` / `read`）として抽象化し、テストではフェイクを注入して外部接続なしに検証する。
  実運用では `redis`/`ioredis` の薄いラッパを渡す。
- メッセージ直列化は `CaptionEvent` の JSON。`isValidCaptionEvent`（shared）で受信時に検証する。
- 既存 `CaptionPipeline` は `CaptionBus` を注入するだけなので、`InProcessCaptionBus` を
  `ValkeyStreamsCaptionBus` に差し替えるだけで分散化できる（呼び出し側の変更不要）。

## 結果（トレードオフ）

- **利点**: 追加インフラ不要・低遅延・イベント単位隔離・順序保証・既存運用（Valkey Serverless,
  ゼロスケール）と整合。インターフェース不変のため移行が容易。
- **欠点/留意**: Valkey はメッセージブローカ専用ではないため、超大規模・長期保持・厳密な
  Exactly-once が要る場合は不向き。その場合は Kinesis/MSK を再検討する（本 ADR の射程外）。
  ストリームのトリミング設定とコンシューマのカーソル管理（At-least-once 配信に伴う重複耐性）を
  Sink 側で考慮する必要がある（字幕は冪等に上書き表示できるため実害は小さい）。

## 未解決（将来の別 ADR）
