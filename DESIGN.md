# ライブ配信プラットフォーム 設計ドキュメント

StreamYard 型 YouTube ライブ配信 Web アプリケーション
AWS / イベント単位スケーリング / 多言語リアルタイム字幕
バージョン 1.0

---

## 1. 概要

StreamYard に相当する YouTube ライブ配信用 Web アプリケーション。配信管理者・イベントモデレーター・登壇者が共同で配信を運営し、スライド投影と登壇者映像の合成、発表者の切り替え制御、QR コードやイベントタイトルのオーバーレイ表示、日本語と英語を中心としたリアルタイム字幕翻訳を行う。最終的な映像と字幕は YouTube Live を通じて一般視聴者へ配信される。

システムは AWS 上に構築する。費用最適化を重視し、配信を行っていない期間はメディア処理層と翻訳層をゼロスケールさせ、常時稼働するのは静的配信・API・データストアといった低コストな制御層のみとする。配信単位でメディア処理リソースをオートスケールさせ、現実的な見込みとして最大 3 つの別イベントを同時並行で配信できる想定だが、アーキテクチャ上の上限ではない (ADR 0008)。

### 1.1 用語

| 用語          | 意味                                                                                                     |
| ------------- | -------------------------------------------------------------------------------------------------------- |
| SFU           | Selective Forwarding Unit。WebRTC の映像・音声を参加者間で中継するメディアサーバー。LiveKit を採用する。 |
| Egress        | SFU 上の複数トラックを 1 本の映像に合成し、RTMP で外部へ送出する処理。                                   |
| 字幕イベント  | 音声から生成される、開始時刻・終了時刻・言語・テキスト・確定フラグを持つ最小単位のデータ。               |
| 字幕バス      | 各エンジンが生成した字幕イベントを共通形式で受け取り、各出力先へ配る内部メッセージ経路。                 |
| 出力先 (Sink) | 字幕バスを購読し、YouTube や独自 API など特定の宛先へ字幕を送る実装。                                    |
| ゼロスケール  | リソースを 0 インスタンスまで縮退させ、利用していない間の課金を最小化すること。                          |

## 2. 要件

### 2.1 機能要件

| ID   | 要件                                                                                                   |
| ---- | ------------------------------------------------------------------------------------------------------ |
| F-1  | 配信管理者（約2名）、イベントモデレーター（約3名）、登壇者（約8名）が同一イベントに参加できる。        |
| F-2  | スライドの投影と登壇者映像を 1 つの配信画面に同時に表示できる。                                        |
| F-3  | スライド投影は、登壇者の画面共有と、事前アップロードした資料をアプリ側で送る方式の両方に対応する。     |
| F-4  | 管理者が、発表中の登壇者とそうでない登壇者を制御できる（画面への出し入れ）。                           |
| F-5  | QR コードおよびイベントタイトルを配信画面に表示できる。                                                |
| F-6  | 配信は YouTube Live で行い、一般視聴者は YouTube から視聴できる。                                      |
| F-7  | 登壇者音声のリアルタイム字幕翻訳を行う。最低限、日本語と英語に対応する。                               |
| F-8  | ASR / 翻訳エンジンを複数の選択肢から差し替えられる（LLM 経路、Amazon Transcribe Streaming 経路など）。 |
| F-9  | 別イベントを同時並行で配信できる (現実的な見込みは最大 3、上限は AWS quota と運用設定の soft cap ADR 0008 D-6)。 |
| F-10 | イベントごとに QR コード・配信素材・各種設定を準備・登録できる。                                       |
| F-11 | 確定字幕を配信プロトコルとして外部へ提供できる（独自字幕配信 API）。                                   |
| F-12 | 管理者は Cognito で認証する。モデレーターと登壇者は招待 URL でアクセスする。                           |

### 2.2 非機能要件

| ID  | 要件                                                                             |
| --- | -------------------------------------------------------------------------------- |
| N-1 | 配信を行っていない期間は、メディア層・翻訳層をゼロスケールし、課金を最小化する。 |
| N-2 | 字幕の遅延は 3 秒以内を目標とする。                                              |
| N-3 | サーバーが必要な場合は AWS を利用する。                                          |
| N-4 | 配信録画および確定字幕は S3 に保存する。                                         |
| N-5 | イベント単位でメディア処理リソースをオートスケールし、イベント間で干渉しない。   |

### 2.3 字幕出力に関する前提

YouTube Live はリアルタイム字幕として 1 トラックのみを受け付ける。このため、常用経路では確定字幕を 1 言語だけ YouTube へ送出する。複数言語を同時に提供したい場合に備えて、独自の字幕配信プロトコル（API）を別経路として用意する。独自 API は普段の YouTube 配信では使用せず、イベント設定で有効化したときのみ起動する。提供範囲はプロトコル（配信エンドポイントの仕様）のみとし、視聴ページや埋め込み部品は利用側で実装する。

## 3. 全体アーキテクチャ

3 つの層に分かれる。常時稼働する制御層、イベント時のみ起動するメディア層、字幕を生成・配信する翻訳・字幕層。制御層以外は配信が行われていない間はゼロスケールする。

### 3.1 制御層（常時稼働・低コスト）

- **S3 + CloudFront**: 管理画面（SPA）の静的ホスティングと CDN 配信。
- **API Gateway + Lambda**: イベント設定 API、発表者制御 API、メディア層の起動・停止のオーケストレーション。リクエスト課金のため非配信時はほぼ無料。
- **DynamoDB**: イベント定義、参加者・ロール、招待トークン、発表状態などのメタデータ。
- **S3（素材・成果物）**: QR コード画像、スライド資料、配信素材、配信録画、確定字幕ファイル。
- **Cognito**: 管理者の認証。モデレーター・登壇者は署名付き招待 URL でアクセス。

### 3.2 メディア層（イベント時のみ起動・並列実行）

イベント開始時に配信単位で独立したメディア処理スタックを起動し、終了時に破棄する。並列数の現実的な見込みは最大 3 だが、アーキテクチャ上は per-event で独立しているため `MAX_PARALLEL_EVENTS` (soft cap) と AWS quota の範囲で並列実行できる (ADR 0008)。

- **SFU (LiveKit)**: 登壇者・モデレーター・管理者の WebRTC 映像音声を中継。画面共有もここで扱う。
- **合成・Egress**: スライド・登壇者映像・QR コード・タイトルを 1 本の映像に合成し、RTMP で YouTube Live へ送出。
- **ElastiCache for Valkey (Serverless)**: ルーム状態、発表者の切り替え状態、低レイテンシな共有状態。

### 3.3 翻訳・字幕層（イベント時のみ起動）

- **字幕パイプライン**: 登壇者音声を分岐して受け取り、ASR と翻訳を行い、字幕イベントを生成する。エンジンは差し替え可能。
- **字幕バス**: 生成された字幕イベントを共通形式で集約し、各出力先へ配る。
- **出力先**: YouTube 字幕トラック（1言語・常用）と、独自字幕配信 API（多言語・任意起動）。

### 3.4 データフロー

1. 登壇者・モデレーター・管理者がブラウザから SFU に接続し、映像音声・画面共有を送る。
2. 管理者が発表者の出し入れを操作すると、状態が Valkey に反映され、合成処理に伝わる。
3. 合成処理がスライド・登壇者・QR・タイトルを 1 枚の映像に合成し、RTMP で YouTube Live へ送出する。
4. 登壇者音声は字幕パイプラインにも分岐され、ASR・翻訳を経て字幕イベントになる。
5. 字幕イベントは字幕バスに集約され、YouTube 字幕トラックへ（1言語）、必要に応じて独自 API へ（多言語）配られる。
6. 一般視聴者は YouTube で映像と字幕を視聴する。録画と確定字幕は S3 に保存される。

## 4. 参加者・ロール・認証

| ロール               | 想定人数 | 認証方式 | 主な権限                                                         |
| -------------------- | -------- | -------- | ---------------------------------------------------------------- |
| 配信管理者           | 約2名    | Cognito  | イベント設定、メディア層の起動・停止、発表者の出し入れ、全体制御 |
| イベントモデレーター | 約3名    | 招待URL  | 進行補助、登壇者の案内、チャット・Q&A の管理                     |
| 登壇者               | 約8名    | 招待URL  | 自分の映像音声・画面共有の送信、スライド送り                     |
| 一般視聴者           | 無制限   | 不要     | YouTube での視聴、字幕言語の選択                                 |

### 4.1 招待 URL の仕組み

モデレーターと登壇者にはアカウントを発行せず、イベント単位の署名付き招待 URL を共有する。URL にはイベント ID・ロール・有効期限を含むトークンを埋め込み、サーバー側で署名検証する。

- トークンはイベントごと・ロールごとに発行し、失効・再発行が可能。
- 有効期限はイベント開催時間に合わせて設定する。
- 管理者のみ Cognito による恒久的なアカウントを持つ。

## 5. 配信画面の合成

配信画面は、登壇者映像・スライド・オーバーレイ要素を 1 本の映像に合成して生成する。合成は SFU 上のトラックを Egress が受け取り、レイアウトを適用したうえで RTMP 出力する。

### 5.1 合成要素

- **登壇者映像**: 発表中としてマークされた登壇者を表示。複数名同時表示にも対応。
- **スライド**: 画面共有、または事前アップロード資料のいずれか。登壇者と並べて表示する。
- **QR コード**: イベント設定で登録した QR をオーバーレイ表示。
- **イベントタイトル**: 下部または上部のテロップとして表示。

### 5.2 スライド投影の二方式

| 方式             | 内容                                                                                              |
| ---------------- | ------------------------------------------------------------------------------------------------- |
| 画面共有         | 登壇者のブラウザから WebRTC の画面共有トラックを送る。任意のアプリの画面を共有できる。            |
| 事前アップロード | PDF・画像を事前に S3 へ登録し、アプリ側でページを送る。共有が不安定な環境でも安定して投影できる。 |

### 5.3 発表者の制御

管理者は管理画面から、各登壇者を「発表中」「待機」に切り替える。状態は Valkey に保持され、合成処理が即座に反映する。

## 6. リアルタイム字幕パイプライン

字幕機能の核は、字幕を共通形式の「字幕イベント」として生成し、変換エンジンと出力先の双方を差し替え可能にする抽象化である。エンジンも出力先も同じ字幕バスを介するため、組み合わせは自由で、後からの追加も容易である。

### 6.1 字幕イベントの共通形式

- 開始時刻・終了時刻（メディアタイムライン基準）
- 言語コード（例: ja, en）
- テキスト
- 確定フラグ（暫定 / 確定）
- 話者・トラック識別子（任意）

### 6.2 差し替え可能なエンジン層

ASR と翻訳のエンジンは共通インターフェースを実装し、イベント設定で選択する。経路ごとに異なるエンジンを割り当てることもできる。

| エンジン                    | 特性                                                                     | 主な用途         |
| --------------------------- | ------------------------------------------------------------------------ | ---------------- |
| Amazon Transcribe Streaming | 低遅延・安定。ストリーミング ASR。翻訳は Amazon Translate と組み合わせ。 | 常用・低遅延経路 |
| LLM 経路                    | ASR+翻訳または翻訳のみ。文脈を考慮し品質が高いが遅延は増えやすい。       | 品質重視経路     |
| 自前 ASR                    | GPU 上で任意モデルを実行。将来的な拡張・特殊要件向け。                   | 拡張用           |

遅延目標 3 秒以内は Transcribe Streaming + Amazon Translate で現実的に達成できる。LLM 経路は品質に優れるが遅延が増えやすいため、YouTube 字幕トラックには Transcribe 経路、品質を優先したい独自 API の一部言語には LLM 経路を割り当てる。

### 6.3 出力先（Sink）

**6.3.1 YouTube 字幕トラック（常用・1言語）**
YouTube Live はリアルタイム字幕として 1 トラックのみ受け付けるため、確定字幕を 1 言語だけ送出する。視聴者はプレーヤー上で字幕の表示・非表示を切り替えられる。

**6.3.2 独自字幕配信 API（多言語・任意起動）**
複数言語を同時に提供する場合に備え、独自の字幕配信プロトコルを提供する。提供範囲はプロトコル（配信エンドポイントの仕様）のみとし、視聴ページや埋め込み部品は利用側で実装する。普段の YouTube 配信では使用しないため、イベント設定で有効化したときのみ起動し、それ以外はゼロスケールする。

- 字幕バスを購読し、確定・暫定の字幕イベントを全対応言語ぶん配信する。
- WebSocket / SSE などのストリーミング配信を想定。仕様とドキュメントを提供し、ページ実装は利用側に委ねる。

### 6.4 字幕の保存

確定字幕は S3 に保存し、イベント後にトランスクリプトや字幕ファイル（SRT/VTT）として利用できるようにする。配信録画も同様に S3 に保存する。

## 7. イベント単位のスケーリングとコスト最適化

メディア層・翻訳層は配信単位で独立して起動・破棄する。複数の別イベントが同時並行する場合、それぞれに独立した SFU・Egress・字幕パイプラインのセットを割り当て、相互に干渉しないようにする。並列数は per-event で独立した構成となっており、`MAX_PARALLEL_EVENTS` の soft cap と AWS quota が上限になる (ADR 0008)。

### 7.1 ライフサイクル

1. 管理者がイベントを開始すると、制御層（Lambda）がそのイベント専用のメディアスタックと字幕パイプラインを起動する。
2. 配信中はイベントごとに独立したリソースで処理する。並列数は `MAX_PARALLEL_EVENTS` (デフォルト 10) と AWS quota の範囲。
3. 管理者がイベントを終了すると、当該スタックを破棄する。
4. 非配信時に稼働するのは制御層（S3/CloudFront・API Gateway/Lambda・DynamoDB・S3・Cognito）のみ。

### 7.2 コスト最小化の方針

| コンポーネント         | コスト方針                                                |
| ---------------------- | --------------------------------------------------------- |
| S3 + CloudFront        | 静的配信。アクセス量に応じた低コスト。                    |
| API Gateway + Lambda   | リクエスト課金。非配信時はほぼ無料。                      |
| DynamoDB               | オンデマンド課金。メタデータのみで小規模。                |
| ElastiCache for Valkey | Serverless を採用し、利用量に応じた課金。イベント時のみ。 |
| SFU / Egress / 字幕    | イベント時のみ起動し終了で破棄。非配信時はゼロスケール。  |
| 独自字幕 API           | イベント設定で有効化した時のみ起動。普段は休止。          |

### 7.3 並列配信の独立性

別イベントはそれぞれ別の管理者・モデレーター・登壇者が参加する独立した配信。SFU・Egress・字幕パイプラインをイベント単位のセットとして起動するため、あるイベントの負荷が他へ波及しない。各セットは個別の YouTube Live 配信先へ送出する。per-event の URL ルーティング (`/join` が events.media.livekitUrl を返す) で、複数イベントを跨いでクライアントが取り違える事故を防ぐ (ADR 0008)。

## 8. イベントごとの準備・設定

管理者は配信前にイベント単位で必要な素材と設定を準備・登録する。設定は DynamoDB に、素材は S3 に保存する。

- イベントタイトル、開催日時。
- QR コード画像（オーバーレイ表示用）。
- 配信素材（背景、ロゴ、テロップなど）。
- スライド資料（事前アップロード方式を使う場合）。
- 登壇者・モデレーターの招待 URL の発行。
- 字幕の対応言語、YouTube へ送出する言語、エンジンの選択。
- 独自字幕配信 API を有効化するか否か。
- YouTube Live の配信先設定。

## 9. AWS 構成まとめ

| 層           | 主なサービス                                           | 稼働形態                            |
| ------------ | ------------------------------------------------------ | ----------------------------------- |
| 制御層       | S3, CloudFront, API Gateway, Lambda, DynamoDB, Cognito | 常時稼働・低コスト                  |
| 共有状態     | ElastiCache for Valkey (Serverless)                    | イベント時のみ                      |
| メディア層   | SFU (LiveKit), 合成・Egress                            | イベント時のみ起動・破棄、並列実行可 (ADR 0008) |
| 翻訳・字幕層 | 字幕パイプライン, 字幕バス, 各エンジン・出力先         | イベント時のみ起動                  |
| 保存         | S3（録画・確定字幕・素材）                             | 常時                                |
| 配信         | YouTube Live                                           | 外部                                |

### 9.1 実装フェーズでの詳細設計事項

- メディアスタックの具体的なコンピュート選択（Fargate / GPU を要する自前 ASR を使う場合は EC2 GPU など）と起動・破棄のオーケストレーション詳細。
- 字幕バスの実装方式（メッセージング基盤）と暫定・確定字幕の同期方式。
- 独自字幕配信 API のプロトコル詳細仕様（メッセージスキーマ、再接続、認証）。
- YouTube Live API 連携（配信開始・字幕送出）の詳細。
- 障害時のフェイルオーバーと、配信途中のリソース再起動方針。
